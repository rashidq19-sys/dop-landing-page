# DSPOps Marketing Landing Page

## What This Is
Marketing/landing page for DSPOps — a SaaS platform for Amazon DSP (Delivery Service Partner) operators. This repo is the public-facing site, not the app itself.

## Tech Stack
- **Frontend:** React 19 + TypeScript, Vite, TailwindCSS v4, shadcn/ui (Radix primitives), Framer Motion for animations, wouter for routing
- **Backend:** Express (Node.js), PostgreSQL via `pg` (raw queries, no ORM) — only used for waitlist email capture
- **Deployment:** Railway (Nixpacks builder)
- **Dev:** Vite dev server on port 3000, Express on port 3001, Vite proxies `/api` to Express

## Project Structure
- `client/src/pages/Home.tsx` — main page, assembles all sections
- `client/src/components/` — one component per landing page section (HeroSection, ProblemsSection, BeforeAfterSection, PricingSection, etc.)
- `server/` — Express server with waitlist + admin API routes
- `server/db.ts` — PostgreSQL pool + waitlist table init
- `shared/` — shared constants

## Key Patterns
- Each landing page section is its own component in `client/src/components/`
- All copy/content is inline in the components (no CMS, no content files)
- Design system: "Clean Logistics Blueprint" — DM Sans font, off-white bg (#FAFAF8), navy text (#0F1B2D), blue brand accents (#2563EB)
- Scroll animations via Framer Motion
- Email capture forms post to `/api/waitlist` with a `source` field tracking which CTA section the signup came from

## Lessons Learned
### Frontend
- `[IMPORTANT]` This is a SPA with no SSR. Per-page meta tags set via `usePageMeta` (client/src/hooks/usePageMeta.ts) work for Google (it renders JS) but NOT for social link previewers (LinkedIn/Slack/WhatsApp parse static HTML only). Static OG/Twitter tags in `client/index.html` are the fallback shown for any shared sub-page link. Proper fix when needed: build-time prerendering for the route list.
- FAQs are defined once in `shared/faqs.ts` and imported by `FAQSection.tsx`. The same 7 FAQs are also duplicated into the `FAQPage` JSON-LD block in `client/index.html` — when FAQs change, update both. (Build-time JSON-LD generation could remove this duplication later.)
- Wouter routing: feature pages (`/features/rota`, `/features/payroll`, `/features/scorecard`), `/blog`, and `/blog/:slug` are registered in `client/src/App.tsx`. Blog posts are registered in `client/src/data/blogPosts.ts` (each post body is a component under `client/src/content/blog/`).
- Navbar anchor links (`#features`, `#pricing`, `#book-demo`) are absolute (`/#features`) so they work from feature/blog pages too — clicking from `/features/rota` will navigate home then scroll to the section.

### SEO
- `[IMPORTANT]` Static SEO files (`robots.txt`, `sitemap.xml`, `og-image.png`) live in `client/public/`. Vite bundles them into `dist/public/` on build, and the Express server (`server/index.ts`) serves them via `express.static(staticPath)` BEFORE the SPA catch-all — so `https://dspops.app/sitemap.xml` returns the XML, not `index.html`.
- When adding new routes that should be indexed, also add them to `client/public/sitemap.xml`.
- The OG image was generated by writing a styled HTML mockup at 1200×630 and screenshotting it with Chrome headless (`chrome --headless --window-size=1200,630 --screenshot=output.png file://...`). Faster than waiting for an image-gen API; document the markup if it ever needs regenerating.
- Google Search Console is a manual setup step — verify the domain, submit `https://dspops.app/sitemap.xml`, monitor coverage and impressions.

### Backend / Database
- `[IMPORTANT]` Database uses raw `pg` queries, not Drizzle or any ORM. The only table is `waitlist`.
- SSL config for Neon pooler: uses `rejectUnauthorized: false` when `sslmode=require` is in the connection string.
